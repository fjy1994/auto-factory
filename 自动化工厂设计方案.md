# 自动化工厂设计方案

> 最后更新: 2026-05-31
> 本文档描述系统的整体架构和业务流程，不涉及具体代码实现和技术细节。

---

## 1. 系统概述

自动化工厂是一套面向 ROM 分支的全自动测试管理系统。系统通过邮件监听 ，自动完成迭代版本跟踪、刷机部署、任务执行到结果汇总的完整闭环。核心围绕"邮件解析  → 迭代版本创建 → 刷机 → 任务执行"的自动化流水线设计。

---

## 2. 核心业务流程

### 2.1 全流程总览

```
构建通知邮件
    ↓
邮件入库 → LLM 解析提取（分支/迭代版本/ROM版本）
    ↓
匹配分支
    ↓
检查设备资源 → 创建刷机任务 → 自动执行刷机
    ↓
刷机成功 → 测试任务创建并执行
    ↓
测试结果回填
```

### 2.2 邮件扫描与解析

1. 监听转测邮件，解析后存入邮件表
2. 定时扫描邮件表，选取未处理或处理失败次数 ≤3 的邮件
3. 将邮件内容发送给 LLM 提取关键信息：
   - 匹配到的分支（关联到已有分支记录）
   - 迭代版本号（如 B25SP1、B26SP2）
   - 所有转测的 ROM 版本列表（如 "ALN-00 207.0.0.25(SP23C00XXX)"）
4. 提取成功 → 更新邮件分析状态为已处理
5. 提取失败 → 递增失败计数，≤3 次下次继续重试，超过则标记为放弃

> 邮件正文可能较大，入库时按需考虑压缩存储。

### 2.3 迭代版本创建

LLM 解析成功后，如果满足以下条件则在版本跟踪页面自动创建迭代版本记录：
- 提取到的分支名称在系统中存在
- 转测 ROM 信息完整

新版本创建后自动执行**版本比较与废弃检查**：

1. 查询同分支下所有已有的迭代版本
2. 按版本号规则（B{数字}SP{数字}）比较大小：先比 B 后数字，再比 SP 后数字，大的为新
3. 对于所有版本号低于新版本的旧版本，按当前状态分别处理：
   - `NO_DEVICE(0)` 或 `QUEUED(1)` → 直接标记为 `OBSOLETED(6)`，不再执行
   - `FLASHING(2)` → 等待刷机流程执行完成，但**不再创建测试任务**，完成后转入 `OBSOLETED(6)`
   - `FLASH_FAILED(3)`、`TESTING(4)`、`COMPLETED(5)` → 不变，不处理

### 2.4 刷机流水线

刷机触发分为**两种场景**：

---

**场景一：新版本创建时触发**

迭代版本创建后立即执行刷机调度：

1. **资源检查** — 查找是否有满足条件的空闲设备
2. **创建刷机任务** — 满足条件则在刷机管理页面创建一个版本维度的刷机任务（FlashingTask），关联对应迭代版本和 ROM 版本
3. **为每台设备创建刷机过程** — 在刷机任务下为每台参与设备创建独立的刷机过程（FlashingProcess），记录该设备的刷机进度
4. **自动开始执行** — 各设备刷机过程按以下步骤顺序执行：
   - 环境检查 — 确认执行机 ADB 连接正常、设备在线
   - 下载 ROM — 从指定地址下载 ROM 包
   - 刷机 — 通过 ADB/刷机工具将 ROM 写入设备
   - 开机引导 — 设备重启后自动跳过初始化设置
   - 连接 WiFi — 配置网络连接
4. **失败处理** — 任一一步执行失败则终止当前刷机流程，等待用户手动重试该步骤或跳过执行

如果设备不满足条件，则版本进入 `NO_DEVICE(0)` 或 `QUEUED(1)` 状态，等待设备空闲时重新触发。

---

**场景二：设备空闲时重新触发**

当设备状态由 busy/offline/error → idle 时，触发调度检查：

1. 按分支维度分组，每个分支**只取最新创建的版本**（按 B{数字}SP{数字} 比较）
2. 检查该最新版本是否满足刷机条件（有匹配的空闲设备、状态为 NO_DEVICE 或 QUEUED）
    - 满足 → 推进到 `FLASHING(2)`，创建刷机任务并开始执行
   - 不满足 → 等待下次设备空闲事件
3. 同一分支下更旧的 NO_DEVICE/QUEUED 版本**不会被重新触发**（已被新版本替代废弃）

---

**刷机执行流程**

每个刷机过程包含 5 个标准步骤，按顺序执行：

1. **环境检查** — 确认执行机 ADB 连接正常、设备在线
2. **下载 ROM** — 从指定地址下载 ROM 包
3. **刷机** — 通过 ADB/刷机工具将 ROM 写入设备
4. **开机引导** — 设备重启后自动跳过初始化设置
5. **连接 WiFi** — 配置网络连接

每步骤独立管理状态（等待中/执行中/成功/失败），支持失败后手动单步重试或全部重试。刷机状态通过前端实时展示，每步骤以时间线形式呈现状态、错误详情和重试次数。

### 2.5 任务执行

刷机成功后自动触发：

1. **创建任务** — 根据该迭代版本所在分支的 TaskDef（任务定义），在任务中心自动创建执行任务
2. **自动开始执行** — 任务创建后立即开始，根据配置的批次大小分批下发用例到 Agent
3. **Agent 执行** — 部署在设备执行机上的 Agent 逐一遍历执行对应的测试脚本
4. **结果回传** — 每个用例执行完成后返回状态（通过/失败）、耗时和错误信息
5. **进度更新** — 后端更新任务进度（百分比），调度下一批用例
6. **回填迭代版本** — 测试结果汇总后回填到迭代版本记录，更新通过率、通过数、失败数

## 3. 系统架构

采用三层架构：

- **前端管理界面** — 提供 Web 管理操作界面，涵盖仪表盘、设备管理、分支管理、刷机管理、版本跟踪、任务中心、用例管理等模块
- **后端服务** — 提供 REST API 和 WebSocket 通信，负责邮件入库与 LLM 解析调度、刷机流程控制、任务调度分发、数据存储
- **设备 Agent** — 部署在执行机上，通过 ADB 控制 Android 设备，执行刷机、开机引导、WiFi 登录、测试脚本等操作

三层之间通过 HTTP API 和 WebSocket 进行通信。

## 4. 各模块功能说明

### 4.1 仪表盘 (Dashboard)

通过图表展示系统整体概况：设备状态分布、任务执行趋势、迭代版本覆盖概览。

### 4.2 设备管理 (Device)

管理接入的 Android 设备，记录设备序列号、机型、当前状态（空闲/忙碌/离线）等信息。支持卡片布局和分页。

**离线检测机制：** 设备（或关联的 Agent）定期向服务端发送心跳报告。服务端记录 `last_report_time`，若该时间超过预设超时阈值（如 3 倍心跳间隔），则自动将设备标记为 offline。当设备重新上报心跳时恢复为 idle 或其他实际工作状态。

### 4.3 分支管理 (Branch)

管理 ROM 分支。分支是顶层概念，例如主干、B26、开发等。每个分支可配置：
- 版本匹配规则（支持正则，用于匹配迭代版本号）
- 邮件标题匹配规则（支持正则，用于匹配构建通知邮件）
- 分支可用/禁用

分支展开后可配置**任务定义（TaskDef）**，定义刷机成功后按顺序执行的自动化任务：
- 任务名称
- 关联的用例集
- 执行脚本路径
- 设备选择方式二选一：
  - **按型号和数量自动匹配**（`auto`）：指定所需设备数量和限制的机型，系统自动从刷机成功的设备中匹配
  - **手动指定设备**（`manual`）：直接从已注册的设备列表中勾选具体设备
- 批次大小

### 4.4 刷机管理 (Flashing)

以刷机任务（FlashingTask）为维度展示，每个迭代版本对应一个刷机任务。展开后显示该版本下各设备的刷机过程（FlashingProcess）。每个刷机过程包含 5 个标准步骤：

1. **环境检查** — 确认执行机和设备就绪
2. **下载 ROM** — 从指定地址下载 ROM 包
3. **刷机** — 通过 ADB/刷机工具将 ROM 写入设备
4. **开机引导** — 设备重启后跳过初始化设置引导
5. **连接 WiFi** — 配置网络连接

每步骤独立管理状态（等待中/执行中/成功/失败），支持失败后手动单步重试或全部重试。刷机过程卡片以时间线形式展示各步骤状态。

### 4.5 版本跟踪 (IterationTrack)

按分支分组展示各迭代版本的测试覆盖情况。顶部统计面板展示已覆盖/覆盖中/未覆盖数量。每个迭代版本卡片展示：
- 测试结果（通过率、通过数、失败数），各任务分别展示
- 关联的刷机任务（用于跳转刷机管理）
- 未覆盖原因分析（无设备、刷机失败等）

版本跟踪与刷机管理和任务中心联动：
- 刷机失败 → 跳转刷机管理定位到对应刷机任务
- 任务行 → 跳转任务中心定位到对应任务

### 4.6 任务中心 (Task)

展示刷机成功后自动触发的执行任务，支持按状态和迭代版本名称筛选。展示任务关联的迭代版本、执行进度、通过/失败统计。任务展开后通过 task_results 表展示每个用例的执行结果（通过/失败标签、耗时、错误信息）。

### 4.7 用例管理 (TestCase)

管理测试用例库。用例按 L0~L4 五级优先级分类，支持按模块和优先级筛选。提供 Excel 批量导入功能，导入时自动识别表头并智能映射优先级。

## 6. API 接口设计

> 通用约定：
> - 所有 URL 以 `/arkweb` 为前缀，RESTful 风格
> - 请求和响应均为 JSON，字段命名使用 snake_case
> - 字段类型与数据库表定义（章节 5）一致
> - 列表接口返回完整数组，前端自行分页
> - 删除操作统一返回 `204 No Content`
> - **非查询接口（POST/PUT/DELETE）** 统一响应格式：`{"code": 0, "message": "success"}`，成功时 code=0；创建类接口额外返回 `"id": <新记录ID>`；失败时返回对应错误码和错误信息

### 6.1 仪表盘

**`GET /arkweb/dashboard/stats`**

系统概况统计，用于仪表盘首页展示。

**响应 200：**

`device_stats` 对象：

| 字段 | 类型 | 说明 |

| busy | integer | 忙碌设备数 |
| offline | integer | 离线设备数 |

`version_coverage` 对象：

| 字段 | 类型 | 说明 |
|------|------|------|
| completed | integer | 测试完成版本数 |
