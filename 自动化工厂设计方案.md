# 自动化工厂系统设计方案

## 一、系统概述

自动化工厂是一个端到端的自动化测试平台，实现从邮件接收到测试报告生成的全流程自动化。

### 1.1 核心业务流程

```
邮件监听 → 邮件解析 → 设备调度 → ROM刷机 → 代码更新 → 任务执行 → 报告生成 → 智能分析
```

### 1.2 系统目标

- 自动化监听Outlook转测邮件
- 智能提取机型和ROM版本信息
- 设备资源智能调度
- 自动化刷机和测试执行
- 测试报告自动生成与智能分析
- 提供人工分析和干预能力

## 四、详细模块设计

### 4.1 邮件监听模块

#### 功能描述
- 持续监听Outlook邮箱
- 识别转测邮件（基于主题/发件人/关键词）
- 提取机型和ROM版本信息
- 下载ROM附件

#### 前端设计原则（隐私保护）
- 不展示任何邮件内容、主题、发件人等敏感信息
- 仅提供邮件服务状态监控、启停控制、配置管理功能
- 所有用户权限一致，无角色区分

#### 核心API功能说明
| 接口 | 功能 | 说明 |
|------|------|------|
| 获取服务状态 | 查看当前正在运行的邮件服务实际状态 | 只读查询，不改变任何配置 |
| 测试连接 | 用用户输入的新配置临时验证连接有效性 | 不影响当前运行服务，用于保存配置前验证 |

**两者区别**：
- **获取服务状态** = 了解现状（看汽车仪表盘，发动机是否在运转）
- **测试连接** = 验证新配置（换电池后先试试能不能打火）

### 4.2 设备管理模块

#### 功能描述
- 设备自动发现与注册
- 设备状态自动更新与同步
- 设备锁定/释放机制

#### 整体架构
采用Agent主动上报模式：
- 部署Agent程序扫描本地USB设备
- Agent定时扫描并向主服务上报设备信息
- 主服务统一管理所有设备的状态和生命周期

#### Agent工作机制
1. **设备发现**：定时扫描本地USB设备，通过ADB命令获取设备完整信息
2. **上报模式**：
   - 全量上报：Agent启动时、每分钟上报所有设备完整信息
   - 增量上报：设备插拔、状态变化时立即上报
3. **远程控制**：接收主服务指令，执行本地操作（设备重启、刷机等）

#### 设备自动注册流程
主服务收到Agent上报后：
1. 按设备序列号匹配：
   - 已存在：更新设备信息、状态、最后上报时间
   - 不存在：自动创建设备记录，初始状态为"空闲"
2. 对比上一次上报，消失的设备自动标记为"离线"
3. 记录所有状态变更历史

#### 设备数据模型
设备核心数据字段：
- 设备序列号（唯一标识）
- ROM版本
- 执行机IP（Agent所在机器的IP地址）
- 浏览器版本（被测浏览器版本号）
- 设备状态（空闲、忙碌、刷机中、离线、异常）
- 备注（支持用户手动编辑）
- 最后上报时间

#### 设备状态定义
| 状态 | 说明 |
|------|------|
| 空闲 | 设备在线，可被调度执行任务 |
| 忙碌 | 设备正在执行测试任务 |
| 刷机中 | 设备正在执行ROM刷机 |
| 离线 | 设备未在Agent上报列表中 |
| 异常 | 设备ADB状态异常 |

#### 状态转换规则
- 设备注册/上报在线 → 空闲
- 任务分配 → 忙碌
- 刷机开始 → 刷机中
- 任务/刷机完成 → 空闲
- 连续未上报 → 离线
- ADB异常 → 异常

#### 设备调度算法
1. 按机型匹配查找空闲设备
2. 优先选择最近上报正常的设备
3. 锁定设备并关联任务
4. 无可用设备时进入等待队列

### 4.3 刷机服务模块

#### 功能描述
- ROM包统一管理与分发
- 刷机任务创建与调度
- 刷机过程实时监控
- 刷机结果校验与状态同步

#### 核心前提
- Agent所在执行机已预置刷机脚本
- 脚本输入参数：设备序列号、ROM版本号
- 脚本内置能力：ROM校验、分区刷写、设备初始化、WiFi连接、自动重启

#### 整体刷机流程
刷机是自动化流程的中间环节，无需人工干预：
1. 解析邮件提取机型和ROM版本号
2. 匹配到可用设备后，自动触发刷机流程
3. 主服务通过WebSocket直接下发刷机指令给对应Agent
4. Agent收到指令后，立即调用本地刷机脚本，传入设备序列号和版本号
5. 脚本内部自动完成ROM下载、校验、刷写、初始化、WiFi连接等全流程
6. 脚本执行过程中，Agent实时捕获脚本输出并上报进度和日志
7. 主服务实时更新主任务状态，前端实时展示刷机日志
8. 脚本执行完成，Agent上报最终结果（成功/失败）
9. 刷机成功则继续执行测试任务，失败则终止流程并标记异常

#### ROM包管理
- 刷机脚本内置ROM下载能力，根据版本号自动从指定源下载
- 主服务只需传递从邮件中提取的版本号，无需维护ROM文件
- 邮件附件中的ROM信息直接透传给脚本

#### 刷机状态
刷机状态内嵌在主测试任务中，无独立刷机任务实体：
| 状态 | 说明 |
|------|------|
| 刷机中 | 脚本正在执行全流程（含ROM下载、刷写、初始化） |
| 成功 | 刷机完成，继续执行测试 |
| 失败 | 刷机过程出错，任务终止 |
| 超时 | 刷机执行超过最大时间，任务终止 |

#### 异常处理机制
- ROM下载失败：自动重试3次，失败则任务标记为失败
- 脚本执行异常：捕获脚本退出码和错误输出，上报主服务
- 刷机后设备未启动：脚本超时自动结束，标记任务失败
- Agent中途失联：任务超时后标记为异常，设备状态待人工确认

### 4.4 版本管理模块

#### 功能描述
- 分支模板管理
- 分支下任务类型配置
- 版本号匹配规则
- 任务自动创建

#### 核心概念模型
采用分支模板 + 任务实例模式：
1. 分支模板只定义该分支需要执行的任务类型配置
2. 刷机完成后，根据版本所属分支自动创建任务实例
3. 设备匹配时使用转测邮件中的机型信息

#### 分支管理
分支核心属性：
- 分支名称（如：开发7.1、主干7.0、商分6.1）
- 版本匹配规则（支持正则表达式，用于从版本号中匹配识别所属分支）
- 禁用开关（禁用后不再自动创建该分支的任务）

#### 分支任务配置
每个分支可配置多个需要自动执行的任务类型：
- 任务名称（如：功能自动化用例、稳定性自动化用例）
- 执行脚本路径
- 设备数量范围（min~max）：
  - min：当空闲设备超过min台时才会创建任务（刷机、批跑）
  - max：最多同时使用max台设备并行批跑
- 用例匹配（支持正则表达式，匹配该任务需要执行的测试用例）

#### 版本匹配与任务自动创建流程
1. 从邮件中提取机型和完整版本号信息
2. 通过版本匹配规则（正则表达式）匹配到所属分支
3. 若分支已禁用，则跳过不创建任务，版本进入待执行队列并标记原因
4. 查找该分支配置的所有任务类型
5. 检查空闲设备是否满足任务配置的min数量要求：
   - 不满足：版本进入待执行队列，标记为"等待设备"
   - 无对应机型ROM：版本进入待执行队列，标记为"等待ROM"
6. 条件满足后，为每个任务类型创建实际的批跑任务实例，最多使用max台设备并行

### 4.5 任务执行模块

#### 功能描述
- 批跑任务调度与管理
- 自动化代码更新
- 测试用例执行
- 执行过程实时监控

#### 任务调度优先级
1. 按分支优先级：主干版本任务 > 开发版本任务
2. 按任务类型：稳定性任务 > 功能测试任务
3. 按创建时间：先到先得

#### 任务状态机
任务生命周期：排队中 → 执行中 → 成功 / 失败 / 异常 / 已取消

#### 任务执行流程
1. 任务创建后进入排队状态
2. 调度器按优先级分配设备
3. 设备锁定，任务开始执行，更新状态为"执行中"
4. 更新对应分支的最新自动化代码
5. 准备测试环境
6. 根据任务的用例匹配规则（正则表达式）筛选需要执行的测试用例
7. 执行过程中实时上报日志和进度
8. 执行完成生成测试报告，报告数据内嵌于任务记录
9. 释放设备，更新任务最终状态

### 4.6 待执行队列模块

#### 功能描述
解决"转测版本已收到但尚未创建任务"的场景，让用户能感知到所有待处理版本。

#### 队列状态定义
| 状态 | 说明 |
|------|------|
| 等待ROM | 有转测版本，但缺少对应机型的ROM包 |
| 等待设备 | 有ROM包，但对应机型空闲设备不足（未达到任务配置的min数量） |
| 待执行 | 条件满足，等待系统调度触发 |

#### 队列操作
- 立即创建：强制为该版本创建任务（跳过等待条件）
- 忽略：放弃该版本，不再自动创建任务

### 4.7 报告与分析模块

#### 功能描述
- 测试报告生成
- 用例失败自动分析
- 失败原因分类
- 人工分析接口

#### 自动分析引擎
1. **日志关键字匹配**：根据预设的失败模式匹配日志内容，分类失败原因（崩溃、网络问题、断言失败、环境问题、ROM问题等）
2. **历史数据对比**：查找历史类似失败案例
3. **截图分析**：对失败时的截图进行分析（可选）
4. **修复建议生成**：根据失败分类生成对应的修复建议

## 八、部署方案

### 8.1 容器化部署方案
采用Docker容器化部署，包含以下服务组件：
- 后端API服务
- 前端Web服务
- PostgreSQL数据库
- Redis缓存服务
- MinIO对象存储服务
- Celery异步任务Worker

各服务通过内部网络通信，数据通过卷持久化存储。
